数据库升级与数据表新增字段(Android)踩坑记录

发表于 2017-10-10 | 分类于 安卓开发相关

一、概述

开发中随着需求与业务逻辑的不断变化，不可避免会遇到升级数据库，修改数据表的问题。一种场景是应用版本升级后，新版本（假设DB_VERSION_CODE=1）应用覆盖安装旧版本（假设DB_VERSION_CODE=2）应用，旧版本应用产生的数据文件（应用数据库、SP等持久化文件）将会被完整保留下来。此时系统检测到新数据库版本号大于旧版本号，将不会重新创建数据库，而是对旧数据库执行升级操作。另一种场景是用户直接是新版应用的“新安装用户”，此时系统会直接新建数据库（DB_VERSION_CODE=2）。针对不同的应用场景总结（踩坑）下来，有很多地方值得注意。分以下几种情况讨论。

相邻版本数据库升级。比如旧版本数据库VERSION_CODE=1，新版本数据库VERSION_CODE=2，那么从旧版本升级到新版本就是相邻版本升级。这种情况下，如果需求对旧版本数据库改动较大，需要存储的字段较多，那么就可以考虑新建一张数据表来单独存储这些字段以避免对原数据表做较大的修改。然而有时需求变化很简单，只需要增加一个字段，那么就没有必要新建数据表了，直接在旧数据表末尾新增一个字段就能解决问题。但是这样就需要同时维护两张数据表，相应的维护成本也增加了。那有没有办法既满足新需求增加字段，又能同时不增加数据表呢，答案是肯定。我们可以将原数据表重命名为一张临时表，然后创建一张同名并带有所有需求新加字段的新数据表，然后将临时表数据复制到新表，最后删除临时表即可。

跨版本升级数据库。如果应用程序发布了多个版本，以致出现了三个以上数据库版本，如何确保所有的用户升级应用后数据库都能用呢？比如旧版本数据库VERSION_CODE=1，最新版本数据库VERSION_CODE=3，那么从旧版本升级到最新版本就是跨版本升级。跨版本升级数据库相对第一种情况较为复杂和棘手，一般做法有两种分析如下。

  • 确定相邻版本的差别，从版本1开始依次迭代更新，先执行V1到V2，再V2到V3…
  • 确定每个升级版本与现版本数据库的差别，为每个case撰写专门的升级代码

方式一的优点是每次更新数据库的时候只需要在onUpgrade方法的末尾加一段从上个版本升级到新版本的代码，易于理解和维护，缺点是当版本变多之后，多次迭代升级可能需要花费不少时间，增加用户等待。方式二的优点则是可以保证每个版本的用户都可以在消耗最少的时间升级到最新的数据库而无需做无用的数据多次转存，缺点是强迫开发者记忆所有版本数据库的完整结构，且每次升级时onUpgrade方法都必须全部重写。接下来结合一个实例来说明一下上述方式一，方式二的实现大同小异。

二、实例

  • 旧版本应用（APP_VERSION_CODE=1）对应的数据库（DB_VERSION_CODE=1）有一张表NOTIFICATION_BACKUP，保存USER_NAME和USER_TYPE两个字段。
  • 中间版本应用（APP_VERSION_CODE=2）对应的数据库（DB_VERSION_CODE=2）在旧版基础上增加了一个字段PRICE。
  • 由于业务逻辑变化巨大，导致最新版本应用（APP_VERSION_CODE=3）对应的数据库（DB_VERSION_CODE=3）需要新增一张表SERVICES_BACKUP来保存SYSTEM_SERVICE和NORMAL_SERVICE两个字段。

我们代码中操作数据库时一般继承SQLiteOpenHelper，重写onCreate()，onUpgrade()，onDowngrade()等几个重要函数。他们分别执行创建、升级、降级数据库的操作。构造函数里面的version参数就是我们定义的版本号，第一次使用数据库时传递的版本参数将被系统记录，并调用SQLiteOpenHelper#onCreate()方法进行建表操作。后续传入的版本号如果大于记录的版本，则会调用SQLiteOpenHelper#onUpgrade()方法升级数据库。

当第一版应用安装时，此时数据库版本号为1，系统会保存当前数据库版本号。直接执行onCreate()函数新建数据表NOTIFICATION_BACKUP，并初始化数据表中的各个字段为它们赋值。

当第二版应用开发时，由于此版本的需求变化较小，所以只需要新增一个字段即可满足业务逻辑的变化。那么如前文分析，没有必要新建一张数据表来单独保存这个字段了。那么只会我们可以修改旧版数据表新增加一个PRICE字段。那么此时就需要对原表执行升级操作。那么如何执行升级操作呢？系统会检查当前数据库版本号（DB_VERSION_CODE=2），如果大于保存版本号（DB_VERSION_CODE=1），就不会再执行onCreate()函数，而是执行onUpgrade()函数升级数据库。显然我们添加新字段的代码应该在onUpgrade()函数中执行。

当第三版应用开发时，由于业务逻辑改版导致原数据库不能满足新需求。此时需要新建一张数据表，同样对原有数据库做了修改，那么就要升级原数据库。所以此时我们修改数据库版本号DB_VERSION_CODE=3。新建数据表的操作同样需要放在onUpgrade()函数中执行。

三、总结

最后总结一下特别需要注意的踩坑点。

  • 重命名表SQL格式为ALTER TABLE <OLD_NAME> RENAME TO <NEW_NAME>;，很明显，增加新表（或者删除没有外键的表）的操作代价很小，只需要在onUpgrade()中写好建表操作和插入初始数据就好了。
  • 表中新增字段SQL格式为ALTER TABLE <TABLE_NAME> ADD <COLUMN_NAME> <COLUMN_TYPE>;，SQLite数库对ALTER TABLE命令支持非常有限，只能在表末尾添加列，不能修改列定义，不能删除已有的列。也就意味着在表末尾追加字段不能对表结构做较大的调整，比如调换某两个字段的位置，在任意位置新加字段等需求。如果要对表结构做较大调整可以通过临时表的方式来解决。具体有三步。
    • 将现有表重命名为临时表，并新建一张同名新表
    • 将临时表的数据导入新表（注意处理修改的列）
    • 删除临时表
  • 导入新数据的INSERT INTO SELECT语句中不能出现values关键字
